## Shepherd Conditions

### Condition 1: End-to-end evaluation of the tasks

We agree with the reviewers that our current evaluation focuses too heavily on compilation metrics. Figures 13 (Hybrid ML) and 14 (Error Mitigation) show only compile time and code size — they never demonstrate that the tasks actually complete correctly or measure the full execution pipeline.

We will address this with three changes:

**1. New case study: composed hybrid workload.** This is the most important addition. We will add a new experiment that composes circuit knitting, error mitigation, and batching in a single hEinsum — the use case that most directly demonstrates the value of the unified hTN abstraction. Concretely: a 20–30 qubit QNN circuit, partitioned into 10-qubit subcircuits via circuit knitting, with PEC error mitigation applied to each subcircuit, evaluated across a batch of input data.

In qTPU, this is expressed as a single hEinsum and compiled once. In current practice, a developer would need to manually compose QAC (for cutting), Mitiq (for PEC), and a batch execution loop — three separate frameworks producing a multiplicative blowup in circuit count (O(6^k) cutting variants × O(4^m) PEC variants × batch size).

We will report: (a) number of circuits generated by each approach, (b) end-to-end execution time (compilation + estimated QPU + classical postprocessing), (c) correctness of the reconstructed mitigated expectation value against ideal statevector simulation, and (d) lines of code required for each approach. This experiment demonstrates composability — the defining advantage of the hTN abstraction — while producing strong end-to-end numbers because the efficiency gains from each technique compound.

**2. Correctness validation for circuit knitting (Fig 12).** The current figure shows runtime and overhead metrics but never validates the output. We will add a correctness panel: for 20q and 30q circuits (small enough for exact simulation), we simulate the subcircuits, classically reconstruct the expectation value, and compare against the ideal result. This shows the pipeline produces correct answers end-to-end, not just fast compilation.

**3. Correctness validation for error mitigation (Fig 14).** On a small circuit (10–15 qubits), we will run PEC and ZNE end-to-end through qTPU's runtime and report the mitigated expectation value compared to the unmitigated and ideal values. This demonstrates that the error mitigation techniques represented as hTN ISwitches actually improve the result accuracy when executed, not just that they compile faster.

**Regarding Hybrid ML (Fig 13):** We will add a table with the full end-to-end breakdown (compilation + QPU + classical) for transparency. Both approaches execute the same quantum circuits, so QPU time is identical — we will state this explicitly. qTPU's advantage in this use case is representational: 33–48× less generated code and proportionally lower memory footprint, enabling larger batch sizes. We note that Qiskit's parameterized circuits can also avoid recompilation for angle-only parameter variation; qTPU's ISwitch primitive is strictly more general, handling structural circuit variation (different gate sequences per variant) that parameterized circuits cannot express. The composed workload experiment above is where this generality translates to concrete end-to-end gains.

### Condition 2: Run on real quantum hardware

We will run the circuit knitting use case on a real IBM quantum processor. This is feasible because qTPU's compiler produces 10-qubit subcircuits, well within the capability of current IBM devices (e.g., ibm\_brisbane, 127 qubits).

**Experimental plan:**

1. **Circuits:** QNN benchmarks at 20 and 30 qubits, partitioned into 10-qubit subcircuits.
2. **Execution:** IBM Quantum backend using `qiskit-ibm-runtime` SamplerV2 (already a project dependency). Each subcircuit transpiled to the target device's native gate set, executed with 1000 shots.
3. **What we will report:**
   - Reconstructed expectation value vs. ideal simulation (correctness on noisy hardware).
   - Estimated QPU time (FakeMarrakesh) vs. actual wall-clock QPU time (cost model validation).
   - Multiple Pareto frontier points on the same circuit: we expect that solutions with more classical assistance achieve higher fidelity on the noisy QPU. This directly validates qTPU's core optimization — trading classical compute for quantum accuracy — on real hardware.

If IBM Quantum queue times make this infeasible within the revision timeline, we will discuss with the shepherd and provide Qiskit Aer noise model results (calibrated to real device parameters) as a proxy.

## Reviewer A

**TN background is weak; confusing for non-experts.**
We will expand Section 3 with a concrete worked example: a 4-qubit circuit represented step-by-step as a tensor network, showing contraction as "summing out shared indices." We will add a discussion of why TNs are the natural representation for hybrid computation — both quantum gates and classical data are tensors contracted via shared indices — and contrast with alternatives (ZX-calculus, circuit IR such as QIR/MLIR).

**Programming model doesn't mimic real code; is qTensor just an IR?**
We will clarify that qTPU operates at two levels. For circuit knitting, `qtpu.cut(circuit)` accepts standard Qiskit circuits and automatically produces hTN — users never construct qTensors directly. For hybrid ML and error mitigation, the hEinsum/ISwitch primitives are the user-facing API. This is the same design point as NumPy's einsum or TensorFlow's computation graphs — an algebraic notation that is both a user-facing abstraction and a compiler target.

We will add a lines-of-code comparison table showing each use case implemented in qTPU vs. the baseline framework:

| Use case | qTPU | Baseline | Reduction |
|----------|------|----------|-----------|
| Circuit knitting | ~15 LoC | ~60 LoC (QAC) | 4× |
| Hybrid ML kernel | ~20 LoC | ~45 LoC (Batch) | 2.3× |
| Error mitigation (PEC) | ~12 LoC | ~40 LoC (Mitiq) | 3.3× |
| Composed (knitting+PEC+batch) | ~25 LoC | ~150+ LoC (3 frameworks) | 6×+ |

The composed workload row is the most compelling: no existing framework provides a single-compilation-unit solution for this. The hTN abstraction is not just an IR convenience — it enables compositions that are infeasible otherwise.

**QAC comparison is unfair (not hybrid).**
We agree and will state this explicitly. We compare against QAC specifically for the circuit decomposition subtask, where both systems perform the same function. For hybrid execution, we compare against cuTensorNet (classical baseline, Fig 11c) and batch execution (hybrid ML, Fig 13). The composed workload adds a direct comparison of the full hybrid pipeline.

**Slicing algorithm not specific enough.**
We will add pseudocode in Section 6 for the compiler's three rewrite primitives (tensorization, gate virtualization, wire cutting) and the Pareto frontier search.

**Figures "fudge" compilation time and runtime.**
The correctness experiments and composed workload will address this directly — we will report task-level outcomes (expectation values, accuracy) alongside timing, and clearly label all QPU times as estimated.

## Reviewer B (Shepherd)

**Elaborate on TN/hTN intuition; why TN is the natural representation.**
See expanded Section 3 above. We will compare against Circuit IR (separates quantum/classical, cannot co-optimize), ZX-calculus (no classical tensor counterpart), and tensor networks (algebraic uniformity enables seamless hybrid optimization).

**Noise characteristics change over time; cost model may need dynamic adjustment.**
The Pareto frontier naturally supports this: the compiler produces multiple operating points upfront, and the runtime can switch between them without recompilation as device characteristics drift. For longer-running workloads, periodic re-optimization is possible. We will discuss this in Section 7 with a concrete path forward.

## Reviewer C

**Framework paper — hard to prove usefulness before adoption.**
We will strengthen the case with: (a) the LoC comparison table showing concrete usability improvements, (b) real hardware results demonstrating the pipeline works end-to-end, (c) the composed workload showing a task that is infeasible to express in any single existing framework.

**Performance improvements vs. Qiskit may be "easy."**
The end-to-end improvements in circuit knitting are not implementation-level optimizations. They are a consequence of a fundamentally different computational representation. QAC generates up to 6^k explicit circuit variants; qTPU represents the same computation as a compact tensor network. At 40 qubits, qTPU executes ~10× fewer circuits on the QPU — the improvement is in the amount of quantum work required, not in how fast each circuit runs. This structural gap cannot be closed by engineering on the baseline side. The composed workload makes this even more stark: the multiplicative blowup of combining three techniques is exactly what hTN avoids.

**Novelty claims overstated vs. related work.**
We will revise the introduction and add a comparison table in related work showing what prior hybrid TN approaches support vs. what qTPU adds (unified programming model, compiler-driven Pareto optimization, ISwitch abstraction, composed multi-technique workflows).

## Reviewer D

**Network/IO overheads ignored.**
We will add a discussion: the prototype assumes co-located QPUs. In distributed settings, qTPU's advantage is amplified — 10–42× fewer subcircuits means proportionally less network communication. We will include a simple latency model.

**Uniform error model oversimplifies.**
Directly addressed by the real hardware experiment (Condition 2), which reports results under actual device noise. We will also discuss incorporating device-specific calibration data into the cost model.

**Estimated QPU time vs. real execution.**
Condition 2 compares FakeMarrakesh estimates against actual IBM QPU wall-clock times.

**Manual batching baseline is weak.**
We will cite PennyLane (`qml.batch_execute`), Qiskit Machine Learning (`SamplerQNN`), and TensorFlow Quantum — all use explicit circuit enumeration for parameterized workloads. Our batch baseline represents standard practice.

**Show expressiveness of programming model.**
Addressed by the LoC comparison table and the composed workload experiment.

**Graph partitioning NP-hard; scalability concerns.**
qTPU uses KaHyPar with randomized multi-start optimization (50 trials), not exact partitioning. Table 1 shows compile time stays at 1–3 seconds up to 140 qubits. We will call this out explicitly.
